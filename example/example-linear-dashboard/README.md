# Example · Linear Dashboard

[![Deploy example to GitHub Pages](https://github.com/Eldergenix/SUPER-DESIGN/actions/workflows/deploy-example.yml/badge.svg)](https://github.com/Eldergenix/SUPER-DESIGN/actions/workflows/deploy-example.yml)

**Live demo:** <https://eldergenix.github.io/SUPER-DESIGN/>

> **A dogfooded Super Design example.** A pixel-faithful recreation of
> the Linear "Projects" view, built end-to-end from the
> `super-design-md/linear.app/DESIGN.md` brand extraction using only
> tokens generated by the skill's `generate-theme.mjs`.

This example exists to answer the question **"how do you actually use
this skill on a real UI?"** — not with a toy button, but with a full
application view that has a sidebar, nav hierarchy, data table, and
chrome elements. Every visual choice in every component traces back to
a token declared in `DESIGN.md`. If you grep the `src/components/`
directory for `#`, `rgb(`, `hsl(`, or `oklch(`, you will find **zero
matches**. That's the point of the skill.

## What it recreates

The Nexis Foundation workspace Projects view from the provided Linear
screenshot:

- Left rail: workspace switcher (NF avatar + "Nexis Founda…"), Inbox,
  My issues, Workspace section (Projects/Views/More), Your teams section
  with the Nexis Foundation-Deve… team expanded and its Projects child
  selected, Try section (Import/Invite/Connect GitHub), and a help
  button in the bottom-left corner.
- Main area: "Projects" title bar, tab strip with an "All projects"
  selected pill and four utility icons on the right (filter, columns,
  split-view, plus), and a data table with six columns (Name, Health,
  Priority, Lead, Target date, Status).
- One project row: **Nexis Homebase** — "Integrate user onboarding to
  track user wallet connections", No updates, ---, No lead, empty
  calendar, 0%.
- Floating bottom-right: Ask Linear button + History icon.

## Run it

From this directory:

```bash
npm install
npm run dev
```

Open <http://localhost:5173>. The app forces `.dark` on `<html>` in
`index.html` so you get Linear's native dark medium — there's no light
theme toggle in this demo.

## Skill workflow demonstrated

This example walks through the full Super Design pipeline from a brand
extraction to running UI. If you follow along with `super-design`'s
7-pass screenshot-to-code workflow, here's where each pass landed:

| Pass | What it produced |
|---|---|
| 1. Layout | `[Sidebar 232px] [flex-1 Main]`. Main stacks `TopBar`, `TabBar`, `ProjectsTable`, with `AskLinearBar` absolutely positioned. See `src/App.tsx`. |
| 2. Color | Every surface/text/border token lifted from `super-design-md/linear.app/DESIGN.md` and translated into the skill's structured token schema in this project's `DESIGN.md`. No per-component color extraction — the brand doc already has them. |
| 3. Typography | Inter Variable + `"cv01", "ss03"` features set globally in `src/base.css`. Weight 510 (`font-medium`) is the default emphasis weight, per DESIGN.md §3. |
| 4. Spacing & components | Tailwind v4 default spacing scale (`--spacing: 0.25rem`) + semantic `inset/stack/inline` roles from DESIGN.md §4. Components are `Sidebar`, `TopBar`, `TabBar`, `ProjectsTable`, `ProjectRow`, etc. |
| 5. Reconcile | Brand extraction values from `super-design-md/linear.app/DESIGN.md` mapped into this project's `DESIGN.md` with no silent invention. The translation added structure (primitive/semantic/component layers) but no new hex values. |
| 6. Code gen | `src/index.css` generated by `npm run theme:generate`. Components reference only the resulting `--color-*`, `--radius-*`, `--shadow-*` utilities. |
| 7. Visual verify | **Out of scope for this commit** — the skill ships a `visual-diff.mjs` script for this, which you'd run after rendering the app with Playwright. See `package.json` scripts if you want to wire it up. |

## Regenerating the theme

If you edit `DESIGN.md` (colors, radii, shadows, motion), regenerate
the CSS:

```bash
npm run theme:generate
```

This runs the skill's `generate-theme.mjs` against `DESIGN.md` and
overwrites `src/index.css`. **Do not edit `src/index.css` by hand** —
any changes will be lost on the next regeneration. Hand-written CSS
belongs in `src/base.css`, which imports `index.css` and layers on
base-level concerns (body, scrollbar, focus rings, forced-colors).

## Validating DESIGN.md

```bash
# Schema lint
npm run theme:lint

# WCAG 2.2 AA contrast check on every semantic color pair
npm run theme:contrast
```

Both scripts shell out to the Super Design skill at
`../../skill/skills/super-design/scripts/` — no duplication.

## Quality scores

```bash
# Score App.tsx (should hit A-grade)
npm run quality:score

# Batch audit every .tsx in src/, sorted worst-first on stderr
npm run quality:audit
```

The skill's `quality-score.sh` grades on LOC, complexity, tokens,
accessibility, responsive, states, and single-responsibility. All
components in this example are small enough and token-compliant enough
to score in the A range — if yours don't, the batch audit will tell
you which ones to refactor first.

## Token-only discipline

This example is a live test of the skill's **"tokens only in component
code"** rule. Run this from the project root and confirm the output is
empty:

```bash
grep -rnE '#[0-9a-fA-F]{3,8}\b|rgba?\s*\(|hsla?\s*\(|oklch\s*\(' src/components src/App.tsx src/icons src/data
```

Every color comes from a `var(--color-*)` via a Tailwind v4 utility
class (`bg-bg`, `text-fg-muted`, `border-border-subtle`) or an inline
`style={{ color: 'var(--color-warning)' }}` for icon tinting. The
inline `style` exception is allowed by the skill's `validate-tokens.sh`
hook because the value is a CSS variable reference, not a literal.

## Known gaps / intentional simplifications

- **No real data layer.** `src/data/projects.ts` returns one hardcoded
  project. A real app would have React Query / tRPC / whatever.
- **No routing.** The "Projects" nav item is hardcoded as selected. No
  `react-router-dom`, no URL state.
- **No keyboard shortcuts.** Linear has a rich command palette — this
  demo is presentation-only.
- **Icons are hand-drawn SVGs.** They approximate Linear's outlined
  icon style but are not pixel-identical to Linear's private icon set
  (which is, understandably, not public). If you want exact fidelity,
  swap `src/icons/index.tsx` for `lucide-react` or an equivalent — every
  icon used here has a lucide counterpart.
- **No font-size tokens in `DESIGN.md`.** The skill's
  `generate-theme.mjs` doesn't currently parse `font.size` blocks, so
  the text scale is declared in `src/base.css` as an extra `@theme`
  block. See the comment there. This is a known gap in the generator
  and would be a nice follow-up PR to the skill.
- **Light mode exists in `DESIGN.md` but is never rendered.** Linear is
  dark-native; toggling is trivial (flip the `dark` class on `<html>`)
  but the demo hardcodes it so the screenshot matches.
- **Dashboard is read-only.** Every button is visually present but has
  no `onClick` handler. The point was to recreate the view, not ship a
  project management app.

## Source of truth

The tokens in this project's `DESIGN.md` are translated from the brand
extraction at
[`super-design-md/linear.app/DESIGN.md`](../../super-design-md/linear.app/DESIGN.md)
in this repository. The brand extraction uses a human-readable prose
format optimized for documentation; this example's `DESIGN.md` uses
the skill's structured token schema so that `generate-theme.mjs` can
parse it. Both files should agree on every value — if you change one,
mirror the change in the other.

## Why this example exists

The top-level repo README explains how to install the Super Design
skill. That tells you the **mechanics** — which files land where, what
hooks are wired up. This example tells you the **shape** of real usage:
what a DESIGN.md actually looks like when it's translated from a brand
extraction, how the generated CSS fits into a modern Vite+React+Tailwind
v4 app, how far you can push "tokens only in component code" before it
gets in your way (answer: indefinitely), and what the resulting UI
looks like.

If you're evaluating whether to adopt the skill on a real project, clone
this one, run `npm install && npm run dev`, and watch the dashboard
render. Then try changing a single token in `DESIGN.md` — say
`--color-accent` — re-run `npm run theme:generate`, and watch every
accent surface in the UI update in one atomic regeneration. That's the
skill's core promise.
